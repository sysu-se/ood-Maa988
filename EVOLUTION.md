# EVOLUTION

本次 Homework 2 的核心变化有两个：Hint 和 Explore。我的实现没有推翻 Homework 1 的对象模型，而是在原来的 `Sudoku` / `Game` 分层上继续演进，让 `Sudoku` 负责规则推导，让 `Game` 负责会话、history 和状态切换。下面按作业要求，直接说明这次设计是怎么变化的。

## 1. 我如何实现提示功能？

提示功能分成两层：一层负责“算出答案”，一层负责“把答案用到当前会话里”。

在 `Sudoku` 里，我实现了候选数和下一步提示的推导能力。`getCandidates(row, col)` 用来返回某个格子的候选数集合，`findNextHint()` 用来从当前棋盘里找出一个可以直接填写的下一步提示。这样，提示所依赖的规则判断都留在领域对象内部，不会散到 UI 组件里。

在 `Game` 里，我再把这些推导结果变成用户能实际执行的动作。`Game` 提供 `getCandidates()`、`getNextHint()` 和 `applyHint()`，由它来决定当前是否能使用提示、提示之后如何写入 history、如何保持当前局面一致。这样做以后，UI 只负责展示和触发，不需要自己拼规则。

## 2. 提示功能更属于 `Sudoku` 还是 `Game`？为什么？

我认为“提示内容的推导”更属于 `Sudoku`，“提示的应用和会话管理”更属于 `Game`。

原因很简单：候选数、下一步可填项、是否冲突，这些都是数独规则本身，应该由 `Sudoku` 负责；但用户是否在当前会话里使用提示、使用后是否进入 history、是否影响探索状态，这些都属于 `Game` 的职责。

所以这次实现不是把提示全部塞进某个组件里，也不是让 `Sudoku` 直接替 UI 做决策，而是让两者协作：`Sudoku` 负责“看懂棋盘”，`Game` 负责“管理这一步怎么落地”。

## 3. 我如何实现探索模式？

我把探索模式实现成 `Game` 创建的一个临时子会话，而不是简单的 UI 临时变量。

进入探索时，`Game` 会基于当前主局面复制出一个独立的探索局面。之后用户在探索里做的每一步填写、撤销和重做，都只作用在这份临时子会话上，不会直接污染主局面。这样，探索既能保留“先试一试”的灵活性，也能保证主局面的状态始终安全。

探索结束时，用户可以选择提交或放弃。提交时，探索中的最终结果会整体合并回主局面；放弃时，探索副本直接丢弃，主局面回到原状。换句话说，探索不是把 UI 上的输入临时藏起来，而是让 `Game` 真正多维护了一段独立的会话状态。

## 4. 主局面与探索局面的关系是什么？

主局面和探索局面是复制关系，不共享可变对象。

我没有让它们共用同一份棋盘数据，而是让探索从主局面克隆出一份独立副本。这样做的好处是，探索中的修改不会直接影响主局面，也不会因为引用共享而出现“回滚后状态被顺手改掉”的问题。

提交时，我不是把探索过程中的每一步重新拼回主 history，而是把探索结束时的盘面状态当作一个整体合并进主局面。放弃时则直接回到主局面，探索副本被丢弃。这个处理方式比较直接，也更符合“临时子会话”的模型。

## 5. 我的 history 结构在本次作业中是否发生了变化？

发生了变化，但没有脱离线性 history 的基本形态。

Homework 1 里，主局面已经有线性的 undo / redo。到了 Homework 2，我在探索模式下又增加了一层局部 history：主局面继续保留自己的线性栈，探索局面也有自己的线性栈，用来支持探索过程中的反复试探。

所以，这次演进不是把 history 改成树状分支，也不是让主 history 直接接收探索中的每一步，而是让探索拥有独立的临时 history，最后再把结果整体合并回主会话。这样既能支持探索，又不会把回退逻辑搞得过于复杂。

## 6. Homework 1 中的哪些设计，在 Homework 2 中暴露出了局限？

这次作业暴露出的局限主要有三点。

第一，只考虑单一主局面，会让后续很难自然地引入临时分支。Homework 2 里的探索模式本质上就是一个短生命周期的分支，如果一开始没有会话层的概念，后面补起来就会比较别扭。

第二，如果提示逻辑过多依赖 UI，规则判断和展示逻辑就会混在一起。这样不仅不好维护，也不利于后续把提示能力复用到探索模式里。

第三，如果 history 只被理解成“单纯记录操作”，而没有把它看成会话管理的一部分，那么探索这种“试错后回到起点”的流程就很难优雅实现。

这次作业让我更明确地意识到，`Game` 不只是记录历史，它还要负责管理当前会话的状态边界。

## 7. 如果重做一次 Homework 1，你会如何修改原设计？

如果重新做 Homework 1，我会更早把边界划清楚。

- `Sudoku` 专注于棋盘规则、候选数、合法性和可推导信息。
- `Game` 专注于会话、history、提示应用和探索分支。
- UI 只负责订阅状态和触发动作，不参与规则推导。

如果一开始就按这个方式分层，Homework 2 的 Hint 和 Explore 就不是“额外补丁”，而是自然的对象演进。后续无论是扩展提示解释、探索独立撤销，还是做更复杂的状态切换，都会更稳。
