# EVOLUTION

Homework 2 在 Homework 1 的基础上新增了 Hint 和 Explore。我的实现没有推翻原来的对象模型，而是继续沿着 `Sudoku` / `Game` 的分层演进：`Sudoku` 继续负责棋盘规则与可推导信息，`Game` 继续负责会话、history、状态切换和探索分支。下面按要求回答这次设计是怎么变化的。

## 1. 你如何实现提示功能？

提示功能分成“推导”和“应用”两层。

在 `Sudoku` 里，我实现了候选数和可推定提示的能力：`getCandidates(row, col)` 用来返回某个格子的候选集合，`findNextHint()` 用来从当前棋盘里找出可以直接填写的一步。这样，所有和数独规则相关的判断都留在领域对象内部。

在 `Game` 里，我把这些推导结果包装成可执行动作：`getCandidates()`、`getNextHint()`、`getHintInfo()` 和 `applyHint()` 都由 `Game` 对外提供。`Game` 负责判断当前局面能不能用提示、提示之后如何写入 history、以及是否影响探索状态。UI 只负责展示按钮和结果，不直接拼规则。

另外，我把提示拆成了更细的语义：

- 候选提示：展示当前格子的候选集合和原因
- 直接填写答案：把当前选中格直接填成正确值

## 2. 你认为提示功能更属于 `Sudoku` 还是 `Game`？为什么？

我认为提示功能是 `Sudoku` 和 `Game` 协作完成的，但职责中心更偏向 `Game`。

原因是：

- 候选数、唯一候选、下一步可推定项，这些本质上都是数独规则，应该由 `Sudoku` 负责
- 是否允许在当前会话里使用提示、使用后怎么写入 history、是否处在探索模式里，这些属于 `Game` 的会话职责

所以更准确地说，`Sudoku` 负责“算”，`Game` 负责“用”。如果把提示完全放在 UI 里，规则会被打散；如果把提示完全塞给 `Sudoku`，会让领域对象承担太多会话管理职责。

## 3. 你如何实现探索模式？

探索模式我实现成 `Game` 内部的一个临时子会话，而不是 UI 里的临时变量。

进入探索时，`Game` 会从当前主局面克隆出一份独立的探索局面 `exploreSession`。之后用户在探索里做的填写、撤销、重做，都只作用在这份子会话上，不会直接污染主局面。

探索失败时，`Game` 会记录失败状态和失败原因，并把当前失败局面的 hash 记下来，用于“记忆”已经失败过的探索路径。这样当用户再次走到同一个失败局面时，系统可以直接提示该路径已经失败。

探索结束时有三种主要动作：

- 回到初态：把探索局面重置回起点
- 保存：把探索局面的最终盘面整体合并回主局面
- 退出探索：放弃探索副本，主局面保持不变

## 4. 主局面与探索局面的关系是什么？

主局面和探索局面是复制关系，不共享可变对象。

我让探索从主局面克隆出一份独立副本，这样探索中的修改不会直接影响主局面，也不会因为引用共享导致回滚污染。这个设计的关键点是“隔离”：主局面始终是稳定基线，探索局面只是临时分支。

提交时，我不是把探索过程中的每一步重新回放到主 history，而是把探索结束时的盘面当作一个整体快照合并回主局面。放弃时则直接丢弃探索副本，主局面回到原来的状态。

## 5. 你的 history 结构在本次作业中是否发生了变化？

发生了变化，但仍然保持线性 history 的基本形态。

Homework 1 里，history 主要服务于主局面的 undo / redo。Homework 2 里，我给探索模式增加了独立的局部 history：

- 主局面继续维护自己的线性 undo / redo 栈
- 探索局面也维护自己的局部 undo / redo 栈
- 提交探索时，只把最终结果作为一个整体快照进入主 history

我没有把 history 改成树状 DAG，也没有把探索过程中的每一步并入主 history。这样的好处是结构简单，而且可以清楚地区分“主局面的历史”和“探索分支的局部历史”。

## 6. Homework 1 中的哪些设计，在 Homework 2 中暴露出了局限？

这次作业让我看到 Homework 1 的几个局限。

第一，如果只按“单一主局面”来设计，就很难自然支持探索这种短生命周期的分支会话。探索本质上不是普通操作，而是一个独立分支。

第二，如果提示逻辑过多依赖 UI，那么规则推导和展示逻辑就会混在一起，后续不容易扩展，也不方便把提示复用到探索模式里。

第三，如果 history 只被理解成“操作列表”，而没有把它当作会话状态的一部分，就很难优雅地表达探索里的试错、回滚和合并。

第四，Homework 1 里如果过早让 store、组件和业务逻辑互相耦合，到了 Homework 2 再拆分会比较痛苦。探索和提示都要求更清晰的领域边界。

## 7. 如果重做一次 Homework 1，你会如何修改原设计？

如果重做 Homework 1，我会更早把对象边界划清楚：

- `Sudoku` 专注于棋盘规则、候选数、合法性、可推导信息
- `Game` 专注于会话、history、提示应用、探索分支和状态切换
- UI 只负责订阅状态和触发动作，不参与规则推导

如果一开始就按这个思路设计，Homework 2 的 Hint 和 Explore 就不是“额外补丁”，而是原有模型的自然扩展。这样后续无论是提示解释、探索独立撤销，还是更复杂的分支状态，都更容易演进。

## 补充
